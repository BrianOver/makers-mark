# P3 core — data-driven class model (binding design)

Status: binding design for the P3-core build (roadmap `2026-07-15-001-roadmap-beyond-v1.md` Phase 3).
Scope decided with Brian 2026-07-16: **class model only**. Companion entity + augment/enchant
layer are deferred — each ships bundled with the add-on that first consumes it (Necromancer/
summoner add-on brings companions; Enchanter add-on brings augments), reviewed together with
its consumer. No zero-consumer hooks in this core (the P2 review lesson).

The move mirrors P1 (Blacksmith-as-data), on the hero side: `HeroRole` (a closed enum baked
into six files) becomes a string-keyed `ClassDefinition` + `ClassRegistry`, with the three
existing roles relocated as byte-identical reference data. Behavior for the three built-ins is
unchanged; the point is that an add-on class becomes one data definition + one registration
line, exactly like an add-on profession.

## Why string class ids, not an enum

The whole add-on-class premise (Necromancer, Magician, …) needs classes an add-on Claude can add
without editing `Contracts/Enums.cs` (deny-list). A string id keyed into a registry is the P1
profession pattern; an enum caps extensibility at the orchestrator. So `Hero.Role` (enum) becomes
`Hero.ClassId` (string): `"vanguard"`, `"striker"`, `"mystic"`.

## Save-format change (owned, safe)

`Hero.Role` → `Hero.ClassId` changes the serialized Hero shape. Safe here: `SaveCodec` is the only
serializer that carries it (`ChronicleCodec` does not serialize role — verified), and the
determinism/save suites compare live-vs-live JSON (no checked-in golden save fixtures to break).
The change is internally consistent under System.Text.Json. Document it in the codec.

## Contract changes (`sim/GameSim/Contracts/` — orchestrator-owned; worker proposes, I land)

- `Hero.Role` (HeroRole) → `Hero.ClassId` (string). Every Hero construction updates.
- `HeroRole` enum in `Enums.cs`: **remove** once no consumer remains (it is not append-only save
  data any more — the string id is). Confirm no serialized int-role fixture depends on it first.

## New module `sim/GameSim/Classes/`

- `ClassDefinition` record (pure data):
  - `Id` (string), `DisplayName` (string).
  - `BaseHp` (int) — starting/recruit MaxHp base (Vanguard 29, Striker 24, Mystic 20 — the current `HeroRoster.BaseHp`).
  - `BaseAttack` (int) — the current `CombatMath.RoleBaseAttack` (Vanguard 4, Striker 6, Mystic 3).
  - `IsAnchor` (bool) — front-line party anchor (`PartyFormation`); Vanguard true, others false.
  - `AllowsShield` (bool) — Vanguard true, others false (the current `Slot==Shield && role!=Vanguard` rule).
  - `MaxItemWeight` (int?) — per-slot weight cap; Mystic 4, others null (unlimited) — the current `MysticMaxWeight`.
  - `ColorRgb` ((int R, int G, int B) or three ints) — presentation hint, pure data so an add-on class
    is self-describing in the UI. Godot reads it; sim never uses it. Values = the current role colors
    (Vanguard steel-blue, Striker crimson, Mystic violet — pull exact values from `HeroSprite.RoleColor`).
- `ClassRegistry` (mirrors `ProfessionRegistry`):
  - `All` — `ImmutableSortedDictionary<string, ClassDefinition>`, `StringComparer.Ordinal`, the 3 built-ins.
  - `RecruitPool` — `ImmutableArray<string>` = `["vanguard","striker","mystic"]` in **exactly** that
    index order. **This is the recruit determinism contract** (see below). Frozen.
  - `TryGet`, `IsRegistered`, `BuiltIn` ids as consts.

## The recruit determinism trap (must hold byte-identical)

`HeroRoster.CreateRecruit` today draws `role = (HeroRole)rng.NextInt(0, 3)` — 0→Vanguard, 1→Striker,
2→Mystic. Golden replays depend on this exact draw. The new code draws
`RecruitPool[rng.NextInt(0, RecruitPool.Length)]` — and `RecruitPool` MUST be the three built-ins in
`vanguard, striker, mystic` order so `NextInt(0, 3)` maps identically. **A registered class is NOT
automatically recruitable** — `RecruitPool` stays the 3 built-ins forever unless a future mechanism
deliberately (and determinism-safely) expands it. New/test classes live in `All` but not `RecruitPool`.

## Generalizations (behavior byte-identical for the 3 built-ins)

- `Heroes/ShoppingAi.cs` — role-fit reads the definition: shield gate = `!definition.AllowsShield`;
  weight gate = `definition.MaxItemWeight is {} cap && item.Stats.Weight > cap`. Pass-reason prose
  must stay exact for the 3 (`"shields don't suit a {name}"`, `"too heavy for a mystic — …"`) —
  `DisplayName` lowercased yields `vanguard`/`striker`/`mystic`, so the strings are unchanged.
  Keep the tests' asserted strings passing.
- `Expedition/CombatMath.cs` — `RoleBaseAttack(role)` → look up `ClassRegistry` by `hero.ClassId`,
  read `BaseAttack`. Still pure, integer-only.
- `Heroes/HeroRoster.cs` — `BaseHp` reads the definition; `StartingSix` sets `ClassId`; `CreateRecruit`
  draws from `RecruitPool`. Starting cast unchanged (same 6, same ids, same class ids, same HP/gold).
- `Heroes/PartyFormation.cs` — anchor rule reads `definition.IsAnchor` instead of `== Vanguard`.
- CLI `Program.cs`, Godot `HeroSprite`/`IconRegistry`/`HeroesPanel` — read `ClassId`; color from
  `ClassDefinition.ColorRgb`. Godot may reference `ClassRegistry` (sim is a godot dependency).

## Conformance harness + extensibility proof

- `sim/GameSim.Tests/Classes/ClassConformanceTests.cs` — parameterized over `ClassRegistry.All`:
  id matches key, non-blank DisplayName, BaseHp/BaseAttack in sane integer bounds, ColorRgb in 0–255,
  `RecruitPool` is a subset of `All` and non-empty.
- Extensibility proof (mirrors P1's 2-profession test that shipped with only blacksmith live): a
  **test-only** 4th `ClassDefinition` registered in a test, asserted to flow through ShoppingAi
  (its shield/weight rules honored), CombatMath (its BaseAttack used), and a constructed Hero —
  WITHOUT touching `RecruitPool`. Proves the add-on shape; no live 4th class in this core.

## Verification (worker runs, I re-run authoritatively)

- Every existing hero/shopping/combat/party/roster/attribution/save/determinism test passes with only
  mechanical `Role`→`ClassId` construction edits (like P1's `UnlockTalentAction` signature change).
- `Category=Balance` bands **byte-identical** — the 3 classes as data reproduce the exact HP, attack,
  shopping, and party behavior, so no seed's world moves.
- Full `dotnet build Game.sln`, fast lane, Balance, engine lane all green.
- Grep-clean: no `HeroRole` reference remains anywhere after the change.

## Not in this core (deferred)

- Companion party-entity — ships with the Necromancer/summoner add-on that consumes it.
- Augment/enchant layer on Item + attribution multi-crafter rule — ships with the Enchanter add-on.
- Necromancer / Magician / Animal Raising / Enchanter content — add-ons.
- Any expansion of `RecruitPool` beyond the 3 built-ins — a separate, determinism-gated decision.
